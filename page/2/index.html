并发条件下的线程安全

  • 竞态条件

    • 定义：多个线程执行顺序不同时,而得到不同的结果
      • 最常见的是延迟初始化，先检查再初始化对象
      • 以及 ++value 操作
        • a:从内存中读取 value 的值
        • b:在 cpu 中进行 value 的值加1
        • c:从 cpu 中往内存中更新 value 的值
    • 竞态条件,是由于多线程环境环境下,共享的某些逻辑块,因为不是原子操作,而在单个线程中会出现状态的不同,进而影响了整个结果

  • 复合操作

    • 解决竟态条件,是把共享的逻辑,变成原子操作,使得这些逻辑执行过程中不会被多线程的调度机制打断,只会在某一个线程中完整的执行完整
    • 复合操作：多个原子操作组合的情况下,仍然不能保证线程安全性,多个原子操作过程中,仍然有竞态条件：最终结果仍然依赖于各个原子操作的配合.

  • 多线程安全

    • 保护竟态条件下,多个连续的逻辑,以原子(不可再分割)的方式执行，一旦逻辑开始,就会执行结束,中间不会被线程调度机制打断,更加没有 context(上下文)

  • 原子性的支持

    • 代码块同步 Synchronized (用锁来保护状态)
      • 内置锁/监视器锁
      • 同步代码块的两个部分
        • 锁对象的引用
        • 被 Synchronized 保护的代码块
      • 同步方法
        • 只要在方法名前加上 Synchronized 就可以
      • 可重入：保证了子类的方法可以调用父类的方法,而不会产生死锁
        • 每个锁关联了使用线程和计数器

共享对象

  • 对象的可见性
    • 重排序
      • 没有进行同步的情况下,编译器和处理器在编译和运行时,可以对执行的顺序进行调整，以适应 cpu 的多级流水操作(乱序执行)
    • 失效数据
      • 没有同步的情况下,共享变量并发读和写时,某一时刻,读到数据可能是旧的数据,此时新的值在 cpu 寄存器中,还没有来的及写回内存中去。
      • 同步原语会保证,用户设置的 new value 被写回内存中时,不会让另外的线程读操作
    • 非原子的 64 位操作
      • long 和 double 类型的 64 位数据,读和写可能被分成两个 32 bit 的操作,这样并发的读写时,会出现读到某个值的32 bit和另外一个值的低32 bit
      • 使用 volatile 或者 lock 来同步操作
  • 可见性与锁及 volatile 变量
    • 对于同一个锁,某个线程获取后,可以完整看到上一个获取到同一个锁的线程的所有操作结果,这就是变量的可见性.这是因为上一个获取锁的线程,实现了所有的同步操作,释放锁后,该锁才能被当前线程获取
    • volatile 变量
      • 访问(get) volatile 变量时,不会加锁,即不会阻塞线程
      • 写入 volatile 变量相当于退出同步块,读取时,相当于进入同步块
      • volatile 变量的可见性,是当 volatile 变量写入时,cpu 会强制更新内存中的值,而非 cache 中的值。

对象的发布与逃逸

  • 对象的发布
    • 使得对象在当前代码作用域外也能够被使用
      • 发布对象的引用
      • 发布对象的内部状态
      • 隐式发布 this 引用
        • 使用内部类时，(构造函数中开辟一个新线程时,这种情况也是属于内部类)内部类可以隐式引用外部类的 this 引用， 当外部类的构造函数还是运行时，通过内部类的 this 引用会指向一个没有构造完成的外部类.
        • 使用工厂方法,将外部类的构造方法进行 private 保护,只能通过工厂方法获取类的实例,这样在工厂方法中,可以保证外部类的正确构造完成
  • 线程封闭
    • 将数据的作用域范围封闭在单个线程中,这样退化到了单线程的模型了
    • volatile 线程封闭
      • 将 volatile 的 write 操作封闭在某个线程中,防止竞态的产生
      • volatile 的多线程并发 read,始终能够读到最新的值
    • 栈封闭
      • 使用基本类型的局部变量,能够保证任何方法都无法获取其引用
    • ThreadLocal 类
      • ThreadLocal 类将发生竞态的共享变量复制一份，保存到本地线程中,实现线程封闭
      • 相对于锁,ThreadLocal 是一种典型的用空间换时间的策略

对象的组合

  • 目的:实现线程安全的类
  • 收集依赖的状态
    • 各种对象之间的组合,造成了对象状态的增加,线程安全的组合,需要整体上状态的的一致性
    • 线程安全：保证各个状态的同步策略
  • 实例封闭
    • Java监视器模式
      • Java 虚拟机中,每个 Object 或者 Class 都通过相应的逻辑关联到一个监视器上
        • 可以通过 Synchronized 关联到对象的监视器上
      • 使用 Class 中的内置私有的锁对象(监视器)来实现同步
        • 相对于其它通过共有方式能够访问的锁对象,无需参与到其它代码作用域的同步策略中去
  • 线程安全委托
    • 对象之间的状态的同步,可以一些线程安全的状态来替换
      • 普通集合 -> 线程安全的集合
      • 普通变量 -> 不可变的变量(final)
        • 可以任意发布不可变变量
      • 发布变量的引用 -> 发布变量的视图
        • 不希望外部线程来改变该变量
    • 委托失效
      • 单个变量虽然是线程安全,但是单个线程安全的变量组合成的对象,不一定是线程安全的

同步容器类

  • 状态封装,共有方法使用 Synchronized 同步，一次只允许单个线程串行访问
    • Vector
    • Hashtable
      • 线程安全,不代表使用的时候就安全,单个方法进行了同步,但是单个安全的方法组合过程中会变得不安全
      • 单个安全的方法组合成一个逻辑,通过 Java 监视器模式,将整个逻辑变成一个原子超做，实现同步
      • 同步容器类是支持客户端加锁的。(使用同步容器类的监视器)
  • 迭代器和ConcurrentModificationException
    • 及时失败的策略
    • hasNext（）/next() 抛出 ConcurrentModificationException 异常
      • 注意隐式的迭代器
        • toString()
        • hashCode()
        • equal()

并发容器类

  • 多个线程并发访问

    • ConcurrentHashMap
      • 使用分段锁(Lock Striping)
      • 迭代器具有弱一致性(Weak Consistent)，并非即使失效
      • size(),isEmpty()返回的计算结果可能是一个过期的值
      • 不能通过加锁来独占,一些需要原子操作的逻辑已经通过内置的方法提供
        • putIfAbsent()
        • removeIfEqual()
        • remove()
        • replace()
    • CopyOnWriteArrayList
    • 写时复制
      • 迭代器指向的底层的一个不可变数组
      • 读(迭代)和写(修改)线程相互不干扰
      • 当前迭代器返回的元素,不会变,不会收到写(修改)线程的干扰

    • ConcurrentLinkedQueue

    • BlocingQueue

    • 阻塞队列

      • 读取时,空则阻塞
      • 写入时,满则阻塞

        • PriorityBlockingQueue

类的构造

Java中类的构造

- 构造函数

  • 构造方法生成类的实例，是最基本的方法。
  • 当构造器参数较多时 ：
    • a: 提供默认的参数，防止用户输入的数据不对,造成对象生成的不完整
      • 提供多个构造器,每一个都在前一个构造器的基础上减少一个默认参数值。
    • b：使用构造器来传入参数,强制用户完整输入(或者构造器填入默认参数)来确保对象相关的属性的完整性
      • 类的内部提供一个静态的 Builder 类,用来逐个设置相关的参数,最后通过一个静态 builder()方法生成外部相关的类的实例。
      • 其实内部的 Builder 的静态类,只是为了能够访问到被 private 控制了的外部类的构造器
    • c：最后一个,通过先生成一个空的 JavaBean 对象,然后在设置相关的属性值，这个将 JavaBean 完整性校验的逻辑交给调用方去检验了.

- 静态工厂方法/工厂方法

  • 使用工厂方法能够对构造器进行拦截(intercept)， 从而决定生成的对象。对已经生成的类进行缓存(cache), 避免了类的频繁调用
  • 使用静态工厂方法,除了以上的好处，还能从整个 app 的维度来管理类的构造,因为 static 类的生命周期与整个 app 的相同。

Services 是一个静态工厂类,Provider 是一个包裹了 Service 的类,主要作用是作为一个实体注册到 Service 中,并且对外部提供 Service 服务类。

为什么需要一个 Provider 类?（item_01）

  • 我的理解是作为一个工厂方法来控制 Service 的生成,而 Services 仅仅相当于一个类的缓存(cache),注册中心的作用.
  • 还有一点符合面向对象的领域原则，Provider 域通过其它上下文专门用来提供 Service 服务的,而 Serivece 仅仅是作为一个 JavaBean 的对象，不具备任何上下文。

单例的生成

  • 直接通过类中的共有方法,访问类加载时生成的实例
  • 通过类对外提供的方法，访问类加载时候生成的实例，相比上一层多了一层抽象
  • 通过枚举值来构造单例。
    • enum 是一个特殊的 class，其构造函数只有在构造枚举值的时候才会调用,并且是私有的;其枚举值是 public static final 的类型，是属于 class的范围，而非实例;
      -综上:使用枚举值来构造单例的意思是：对于一个 enum 类的某个 enum value(enum 和 enumValue 是一对多的关系)，其构造函数是自动调用，生成一个实例,外界也无法重新调用生成新的值的， 这样就生成了一个单例了。

避免无意义的类的构造

  • 频繁的类的构造,主要是发生在局部变量的构造中
  • 对于 class 中通用的对象,可以将其作用域通过关键字 static 提升到 class 中，在各个对象中共享
  • java 中除了基本值外,一切皆对象,long(value) 和 Long(Object) 是不同的。

类中通用的方法

  • equal()
    • 最好不要覆盖 equal() 方法
    • equal() 表示的是对象的相等,即内存地址相等,而非 value 值相等
  • hashCode()
    • hashCOde 是以对象内存地址的值为参数计算出来,这样能够保证任意一个对象的 hashCode 的值不相等,这样能够同 hashCode 的值做索引,快速找到对象（eg：hashMap）。
    • 既然hashCode与内存地址相关,那么就与 equal()方法相关
    • 覆盖 equal() 的同时,一定要覆盖 hashCode (item_09)
      • 可能出现的情况是 equal() 相等,但是 hashCode 不相等,这样通过索引找对象时,会出现找不到的情况。
  • clone()
    • Cloneable::clone()
    • 注意区分深copy(对象的复制)和浅copy(值的复制)
      • Cloneable::clone() 接口仅仅执行的浅copy,当自定义的Object包含基本数据类型和其它对象类型时,clone()时需要注意：super.clone() 只是负责浅 copy,其它对象类型需要根据需求进行手动的 copy。
      • 从某种程度上,可以理解,Cloneable::clone() 只能进行浅copy，应为不知道你自己的定义的对象类型，就像c++的,拷贝构造函数一样,默认的仅仅执行的浅 copy
  • Comparable 接口
    • Comparable::compareTo() 最主要是和范型算法配合使用
    • 排序算法(eg:sort())会根据compareTo()的结果来进行排序,决定前后两个元素。
      • 返回 1,在 sort() 的角度看上去是 left < right
      • 返回-1,在 sort() 的角度看上去是 left > right
      • 返回 0,在 sort() 的角度看上去是 left == right
      • sort() 总是按照升序来排序，想要实现 left < right，所以我们可以决定整个排序的顺序

范型

  • 原生范型与范型特化
    • 范型在程序运行时,是被擦除的,任何范型都被擦除成了 Object 类型
    • 使用原生范型时(eg:set,list 等等),会跳过编译时的类型检查,在代码运行期间容易产生类型转换(cast)的异常
    • 使用参数化后的范型(eg:List)，代码编译时,由类型检查,类型转换不兼容的异常会在编译器期间抛出来,而非在运行时才发现
  • 无限制通配符与有限类型通配符
    • 使用通配符类型时,表明你不关心或者不知道程序具体的类型信息,而专注于一个通用的代码逻辑
    • 无限类型通配符，想较于原生类型,其仍然有类型安全的特性，因为除了 null,不能将任何元素放入到集合类型中
      • 使用无限类型通配符时,一般是用来比较元素是否为 null.因为使用了 ? 后，只能往里面写入 null 对象了。

枚举

  • 枚举是一个特殊的 class，能够由自己的方法,自己的作用域
  • 枚举与普通类的区别
    • 枚举的实例对象，不能在枚举类外,通过构造函数生成,必须在枚举类中定义生成
    • 枚举的实例对象的个数,是可控的,因为在枚举类的里面,控制了实例对象的生成
    • 枚举值(实例,对象)的名称，即实例(对象)通过构造函数生成时的，自定义的名称(字符串),同时也是在程序中通过枚举类直接引用的那个名称(eg:Color.RED，这个枚举对象的名称就是RED)
  • 枚举的方法
    • values():返回枚举类的所有实例(对象)
    • EnumSet.of():返回一个枚举值的集合

方法

  • 必要性的保护
    • 保护性拷贝,防止出现类中属性的引用被利用
      • get()方法：不要将类中的属性的引用暴露出来,提供给外部的只能是属性的值(value)
      • set()方法: 设置的只能是属性的值(value)，而不能直接将外部的引用设置到类的属性中去
  • 方法重载与覆盖(@Override)
    • 重载方法是在编译时,由编译器基于编译参数而决定的
    • 重载方法是函数名相同,参数不同
    • 方法覆盖是在运行时决定的
    • 方法覆盖函数名相同,参数不同

序列化

  • 序列化是将一个对象编码成字节流，在不同的计算机上进行传输,能够实现跨语言之间的对象交流(反序列化)

ssh登录

ssh(Secure Shell)协议主要用于互联网上计算机的远程登录，为计算机的 Shell(壳层,主要与OS的内核通讯) 层，提供安全的数据传输和使用环境。

安全性如何保证

  • 基于密码的安全验证

    用户请求登录远程主机时,主机将自己的公钥发送给用户，用户的传输数据使用远程主机的公钥进行加密，只有远程主机的私钥才能解密，以此来实现数据的传输安全。

    中间人攻击

    基于密码的验证，当网络上有人截获你的登录请求时,冒充远程主机，发送伪造的公钥给你，(不同于 https，使用的公钥是由与证书中心 CA 公证的)使用伪造的公钥加密通讯数据后,那么会截获并解析你的传输数据,再用截获的登录密码登录远程主机。

  • 基于密钥的安全验证

    既然没有证书中心 CA 确认的公钥，那么我们自己可以在我们的范围内确认某一个公钥(某个实体的身份)的正确性，那就是我们自己生成一对私钥和公钥，通过将自己生成公钥放在远程服务器端。用户请求登录过程中,远程主机为了确认用户的身份，发送一段随机的字符串,让用户用私钥加密后，回传给远程服务端,远程服务端用事先用户存储的公钥进行解密,如果成功了，就证明用户是可信的,直接允许进行登录，而不用输入密码。

    基于密钥的安全登录，不用输入登录名和密码,那么登录时，我们是以那个身份登录的呢？

    生成私钥和公钥的时候，会自动将我的的身份带入进去。ssh-keygen -C “yourName@hostname.com”,如果不设置，默认就是以当前的身份生成该私钥和公钥，公钥文件的最后会有这个用户身份。注意确认唯一的身份
    是 yourName + hostname，不要将 -C 后面的内容想象成电子邮件(email),那是你身份的标识，即:hostname 上的 yourName 用户。

  • 总结

    由于使用 ssh 协议传输的过程中,没有一个统一的公钥证书中心(CA) 来标识网络上的某个实体的身份,我们唯一能确认的就是我们自己生成的公钥,基于密钥的安全验证，实际上是由我们来验证服务端(放置了可信的公钥的实体)，而基于密码的验证方式远程服务端来验证我们,我们的登录请求可能会发往网络上的任何一台计算机,所以可能导致了冒牌远程服务端的出现(不进行任何的验证，只是希望得到我们的登录请求)。
