ssh登录

ssh(Secure Shell)协议主要用于互联网上计算机的远程登录，为计算机的 Shell(壳层,主要与OS的内核通讯) 层，提供安全的数据传输和使用环境。

安全性如何保证

  • 基于密码的安全验证

    用户请求登录远程主机时,主机将自己的公钥发送给用户，用户的传输数据使用远程主机的公钥进行加密，只有远程主机的私钥才能解密，以此来实现数据的传输安全。

    中间人攻击

    基于密码的验证，当网络上有人截获你的登录请求时,冒充远程主机，发送伪造的公钥给你，(不同于 https，使用的公钥是由与证书中心 CA 公证的)使用伪造的公钥加密通讯数据后,那么会截获并解析你的传输数据,再用截获的登录密码登录远程主机。

  • 基于密钥的安全验证

    既然没有证书中心 CA 确认的公钥，那么我们自己可以在我们的范围内确认某一个公钥(某个实体的身份)的正确性，那就是我们自己生成一对私钥和公钥，通过将自己生成公钥放在远程服务器端。用户请求登录过程中,远程主机为了确认用户的身份，发送一段随机的字符串,让用户用私钥加密后，回传给远程服务端,远程服务端用事先用户存储的公钥进行解密,如果成功了，就证明用户是可信的,直接允许进行登录，而不用输入密码。

    基于密钥的安全登录，不用输入登录名和密码,那么登录时，我们是以那个身份登录的呢？

    生成私钥和公钥的时候，会自动将我的的身份带入进去。ssh-keygen -C “yourName@hostname.com”,如果不设置，默认就是以当前的身份生成该私钥和公钥，公钥文件的最后会有这个用户身份。注意确认唯一的身份
    是 yourName + hostname，不要将 -C 后面的内容想象成电子邮件(email),那是你身份的标识，即:hostname 上的 yourName 用户。

  • 总结

    由于使用 ssh 协议传输的过程中,没有一个统一的公钥证书中心(CA) 来标识网络上的某个实体的身份,我们唯一能确认的就是我们自己生成的公钥,基于密钥的安全验证，实际上是由我们来验证服务端(放置了可信的公钥的实体)，而基于密码的验证方式远程服务端来验证我们,我们的登录请求可能会发往网络上的任何一台计算机,所以可能导致了冒牌远程服务端的出现(不进行任何的验证，只是希望得到我们的登录请求)。
