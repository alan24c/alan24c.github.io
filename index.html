Java内存模型

Veröffentlicht am 2017-06-14

Java内存模型(JMM)

Java内存模型解决的问题
  • JMM主要解决的是共享变量何时可见的问题
    • JMM抽象出两种内存
      • 本地内存
      • 主内存
    • 共享变量存在于主内存中,每个线程的本地内存保存着共享变量的读写副本
  • 多线程环境下,一个普通变量的读和写可能在不同线程中进行,由于(编译期间或者指令运行期间的)重排序,可能会引起变量的读写顺序不定，重而导致了变量的不可见问题
重排序引起变量可见性
  • 重排序
    • 编译器优化导致重排序
      • JMM 通过禁止特定类型的重排序,控制编译器生成代码
    • CPU多级流水线指令的并行运行导致重排序
      • JMM 通过在编译时插入内存屏障指令禁止并行运行时的重排序
  • 数据依赖性
名称 例子
写后读 a = 1; b = a
读后写 a = b; b = 1
写后写 a = 1; a = 2

上述 3 种情况不论是编译器重排序或者是执行时重排序后,都会引起数据不一致性

变量不及时刷新回缓存引起变量的可见性
  • 内存屏障指令会强制将CPU缓存中的变量刷新回缓存
JMM内存模型

  • 两个重要的语义

    • happens-before
      • volatile 变量: volatile 变量的写 happens-before 读
      • lock: lock 的解锁 happens-before 加锁
    • as-if-serial
      • 对于存在数据依赖性的操作,不论是编译器还是处理器都不会对这些操作进行重排序
      • 表现形式：程序在单线程情况下,是顺序执行的

  • 多线程情况下 as-if-serial 语义失效

final 语义

final 变量都会在构造函数中被完整初始化，通过构造函数对 final 变量的读取得到都都是完整的对象

  • 构造函数内对一个 final 变量的写入，与随后把这个 final 的引用赋值给别的变量之间不能重排序
    • 保证 final 变量在被引用之前已经构造完毕,不会被重排序到构造函数之外初始化,普通变量不具备这个保证
  • 初次读取一个包含 final 域的引用的对象,与随后读取这个 final 域,这两个操作不能重排序
    • 保证读取一个 final 变量的引用之前会首先读取这个 final 变量的对象引用，普通变量不具备这个保证,可能会首先读取这个普通变量,而后读取包含这个普通变量的对象的引用，此时这个普通变量可能没有被初始化
  • 构造函数内对 final 引用对象的写入,与随后把这个 final 变量的引用赋值给别的变量之间不能重排序
    • 保证 final 引用对象在构造函数中被完整构造

Mysql中的事务

Veröffentlicht am 2017-05-19

事务

什么是事务？

  • 事务是 mysql 执行的一系列 sql语句,可以只包含一条 sql 语句,也可以包含多条复杂的语句。是访问更新数据库中数据项的一个执行单元。

事务的ACID特性

  • A：Atomic，原子性
  • C：Consistency,一致性
  • I：Isolation,隔离性
  • D：Durability,持久性

事务的commit(提交)和rollback(回滚)来保证,一个事务中的 sql 语句要么全部执行成功,若事务中有一条语句执行失败,则全部回滚，保证事务的原子性;同时保证了数据库中数据的一致性,不会出现事务进行过程中的一个中间状态;commit(提交)后的事务,则永久存储在数据库中了，即持久性。

事务的一致性
  • 事务的隔离级别
    • read uncommitted
      • 并发情况下,所有的事务都能看到其它事务未提交的结果
    • read committed
      • 并发情况下,所有的事务只能看到其它事务以提交的结果
      • 这种隔离级别下会出现不可重复读的现象,即,并发情况下,由于其它事务提交了数据的变更，造成了同一个事务,多次读取相同数据时,可能会得到不一样的结果。这违反了事务的隔离性。
    • repeatable read (mysql 默认的隔离级别)
      • 并发情况下,同一个事务中，多次读取相同数据时,会得到同样的结果。
    • serializable
      • 串行化读,对数据没有竞争关系
一致性非锁定读
  • 使用 MVCC(多版本并发控制)来解决并发条件下,读数据时，不采用等待写锁释放的条件下,对读与写操作的控制
    • 如果读的某行数据正在被更改,那么该读取操作不会等待该行上锁的释放，而是去读取行的一个快照(snap)信息。
    • 快照版本的选取
      • read committed 隔离级别下，选取锁定行的最新的一个快照信息 –> 不可重复读的现象
      • repeatable read 隔离级别下,选取锁定行的事务开始前的行的版本信息 –> 可重复读
一致性锁定读
  • 读取数据时,显示对数据库进行加锁,用以保证数据逻辑的一致性
    • 行锁的3种算法
      • Record Lock :单行上锁
      • Gap Lock: 间隙锁
      • Next-key Lock (Record Lock + Gap Lock)
        • 使用 Next-key Lock 解决幻读现象：同一事务下,连续两次同样的SQL语句,可能导致不同的结果,第二次的 SQL 可能会返回之前不存在的行。
幻读，可/不可重复读
  • 幻读：整个表结构而言,整个表结构可能在某个事务进行过程中被更改(数据行的增/删)
  • 可/不可重复读: 对单行数据而言,读取的单行数据的快照版本信息的不同

Mysql中的事务

Veröffentlicht am 2017-05-19

事务

什么是事务？

  • 事务是 mysql 执行的一系列 sql语句,可以只包含一条 sql 语句,也可以包含多条复杂的语句。是访问更新数据库中数据项的一个执行单元。
  • 一个事务的执行要么所有语句全部执行成功(事务 commit),事务中任何一条语句执行失败后,事务则撤销全部改动(事务 rollback)

事务的特性

  • 事务的ACID特性
    • A：Atomic，原子性
      • 事务尽管由多条 sql 语句组成,但是其是一个不可分割的整体,要么全部执行成功,要么什么都不发生。
    • C：Consistency,一致性
      • 事务必须使得数据库从一个一致性状态变为另外一个一致性状态
      • eg:银行转账的例子,事务中：A账户扣钱到B账户,A账户钱减少,B账户钱增加,满足数据库中数据的一致性,该转账事务执行后,不能使得数据库中A账户钱减少了,而B账户的前没有增加。(这种状态是事务进行过程中的一个中间状态)
    • I：Isolation,隔离性
      • 并发情况下,每个事务读写的对象,在该事务提交前都对其它事务不可见。
      • 通常使用锁来实现事务的隔离性。
    • D：Durability,持久性
      • 事务一旦提交了,其对数据进行的更改就是永久性的了。
事务的一致性
  • 事务的隔离级别
    • read uncommitted
      • 并发情况下,所有的事务都能看到其它事务未提交的结果。脏读。
      • 这种隔离级别，不满足事务的隔离性。
        • 脏数据:是指事务对缓冲池中断额行记录做了修改,没有被提交(committed)
    • read committed
      • 并发情况下,所有的事务只能看到其它事务以提交的结果
      • 这种隔离级别下会出现不可重复读的现象,即,并发情况下,由于其它事务提交了数据的变更，造成了同一个事务,多次读取相同数据时,可能会得到不一样的结果。这违反了事务的隔离性。
    • repeatable read (mysql 默认的隔离级别)
      • 并发情况下,同一个事务中，多次读取相同数据时,会得到同样的结果。
      • 解决了不可重复读的现象
    • serializable
      • 串行化读,对数据没有竞争关系
Mysql 并发的读写的控制
一致性非锁定读
  • 使用 MVCC(多版本并发控制)来解决并发条件下,读写冲突。读数据时，不用等待写锁释放。
    • 如果读的某行数据正在被更改,那么该读取操作不会等待该行上锁的释放，而是去读取行的一个快照(snap)信息。
    • 快照版本的选取
      • read committed 隔离级别下，选取锁定行的最新的一个快照信息 –> 不可重复读的现象
      • repeatable read 隔离级别下,选取锁定行的事务开始前的行的版本信息 –> 可重复读
一致性锁定读
  • 读取数据时,显示对数据库进行加锁,用以保证数据逻辑的一致性
    • 加锁类型：
      select ... for update            // 显示加排它锁(X)
select ... lock in share mode    // 显示加共享锁(S)
Mysql 并发控制中的锁

  • 锁的类型

    • 单行锁
      • X：排它锁 (类似Java中的写锁)
      • S：共享锁 (类似Java中的读锁)
        • X 锁与任何锁都不兼容
        • S 锁只与 S 锁兼容
    • 表级锁
      • 读锁
      • 写锁
    • 意向锁
      • IX ：意向排它锁
      • IS ：意向共享锁
      • 意向锁存在的意义：
        • mysql支持多粒度的锁定,运行在行级别上的锁和表级别上的锁同时存在.当事务A申请表的写锁时，需要判断表上是否有其它共享锁或排它锁,这需要遍历每一行。当添加意向锁后,事务B申请 S 锁时,先申请 IS，这样事务A，申请表级写锁时,就会被阻塞,因为表上已经有了 IS 锁,而不用去遍历没一行。

  • 锁的算法

    • 行锁的3种算法：

      • Record Lock :单行上锁
        • 锁住索引,如果建表时，没有索引,则使用主键来锁定
      • Gap Lock: 间隙锁
        • 锁定的是一个范围，该范围不包括记录本身(开区间)

      • Next-key Lock (Record Lock + Gap Lock)

        • 包括记录本身(闭区间)

      • eg:表中由，10,20 这两个索引
        gap lock：(-∞,10),（10,20),(20,+∞)
        Next-key Lock ：(-∞,10],（10,20],(20,+∞)

    • 锁算法的使用

      • 聚集索引时,由 Next-key Lock 降级为 Record Lock，从而提高并发性

      • 聚集索引+辅助索引：聚集索引上为 Record Lock，辅助索引上为 Next-key Lock,同时对辅助索引的下一个索引加 gap lock。

        • 使用 Next-key Lock 解决幻读现象：同一事务下,连续两次同样的SQL语句,可能导致不同的结果,第二次的 SQL 可能会返回之前不存在的行。

          这种情况主要发生在辅助索引上,因为在辅助索引上，不加锁时允许运行时擦入相同的值。而聚集索引是被单行锁定的，同时聚集索引上不会允许擦入相同的值的，这是数据库的一致性，保证聚集索引的唯一性，同时保证不会产生幻读现象。

幻读，可/不可重复读
  • 幻读：整个表结构而言,整个表结构可能在某个事务进行过程中被更改(数据行的增/删)
  • 可/不可重复读: 对单行数据而言,读取的单行数据的快照版本信息的不同

Java反射的基础 : Class对象
  • Java 虚拟机编译时候会创建 class 文件,包含了类的所有信息
    • Class.forName(“nameOfClass”) 加载 class 文件,返回一个 Class 对象
      • Object instanceof “nameOfClass” 和 class.isInstance(Object)的区别
        • 两者动态等价,都是判断运行中的某个对象是否是指定 Class 的实例,语义是:你(Object)是这个类还是从这个类(Class)派生出去的
        • 两者的区别
          • instanceof 关键字
          • isInstance() Class 对象的方法
      • Class 对象的 equale 和 ==
        • equal 和 == 动态等价
      • equale/== 和 instanceof/isInstance() 的区别
        • equale/== 意义一样,instanceof/isInstance() 意义一样
        • equale/== 实际比较的是 class 的对象,不牵涉到继承的问题
        • instanceof/isInstance() 语义是,你是这个类还是从这个类派生出去的 Object(对象)
    • Class 的其它方法
      • Class::getInterfaces() // 返回对象继承的接口(数组)信息
      • Class::getSuperclass() // 返回对象的父类(Java 只允许单一继承,所以只返回一个 Class 对象)
      • Class::newInstance() // 尽管我不知道你的准确的类型是什么,可能是该 Class 下的一个子类,但是请你准确的构造自己
      • Class::getMethods() // 返回对象的方法(数组)信息
      • Class::getConstructors() // 返回对象的构造器(数组)信息
Java反射的作用

  • 利用 class 信息,能够动态生成一个实例,避免了在代码中硬编码 Object 信息

    • Java 中的 SPI

      在代码中定义服务接口,不同的服务提供者提供服务的具体实现,在 META-INF 文件夹下面配置一个与服务接口相同名字的文件,里面定义服务的具体实现类，代码中通过类加载器(ClassLoader)加载
      配置文件中定义的实现类 Class 对象,来完成实例化，而不必事先知道服务的具体实现。

      • dubbo 中,广泛应用了Java 的 SPI 机制.ExtensionLoder 类充当了加载各种插件(Class)的作用
      • 有点类示于C++中的RTTI,不过 C++ 中是需要在代码里显示的初始化实现类的,这样就在代码里面形成了硬编码,当要替换实现时,会要替换很多地方,而Java中的SPI机制，仅仅只是需要更换配置文件中的实现类即可。

Spring中Bean的生成和定义

Veröffentlicht am 2017-05-13

bean的定义

  • 通过 java 代码定义 bean

    • @Configuration
      • 语义：配置类,在该类中包含了在应用中创建相关 bean 的逻辑,可以通过该类管理整个应用中的 bean
      • @Bean: 显示定义 bean
        • 语义:返回一个对象,并且将该对象注册到 spring 上下文中
        • 默认情况下创建的 bean 的 name(id) 就是是方法名
        • (name = “beanName”) 自定义 bean 的名字
      • @Component + @ComponentScan / xml文件配置包扫描 ：通过组件扫描来发现 bean
        • @Component ： 组件(bean)的定义
          • 语义:将被注解的类注册为一个bean，默认的 bean 的名为类名(第一个字母小写)
          • @Component(“beanName”) 自定义 beanName
        • @ComponentScan 开启组件扫描
          • 扫描当前类所在的包中的组件
          • @ComponentScan(basePackages=”nameOfPackages”)/@ComponentScan(“nameOfPackages”) 指定扫描基础包,解决代码重构后,默认的扫描类位置的变动,而造成的扫描包的变动
        • 使用 xml配置文件，通过标签< context:component-scan basa-package = “nameOfPackages”>同样能实现 @ComponentScan 注解的作用

  • xml 文件配置 bean

    • bean的配置文件同样需要遵循相关的规范,以 < beans> < /beans> 为根元素，类似于 @Configuration 的作用
    • 在< bean> < /bean> 显示配置相关的 bean

bean的注入

  • 显示配置完 bean 后,当 bean 初始化时,需要构造/初始化内部相关的引用属性，这个时候我们需要配置各个属性,为各个属性注入相关的引用,当然对于基本变量,这些就是由 spring 自动初始化的

  • 通过 java 代码

    • @Autowired (Spring中的注解)
      • 按照 type 来注入
      • required 属性:标注该注入的对象是否允许为 null
      • @Qualifier(“beanName”) 限制注入的 bean 的名称，当按照 type 来注入的时候,有多个相同 type 的 bean 时,就会出现歧义,通过指定相关的 name(ID)消除这种歧义
    • @Resource （J2EE中的注解）
      • 按照 name 来注入
      • name 属性:限制注入的 bean 的名称(ID)
  • xml文件中指定
    • < constructor-arg > 元素
    • c-命名空间
